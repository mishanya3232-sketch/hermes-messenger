# Безопасность Hermes Messenger

## Главное правило

Frontend не должен знать:

- токены Hermes;
- API-ключи;
- секреты backend;
- админские ключи;
- пароли пользователей.

## Правильная схема

```txt
Frontend → Backend → Hermes
```

Неправильная схема:

```txt
Frontend → Hermes напрямую с токеном
```

## Авторизация

Для MVP:

- логин;
- пароль;
- JWT-сессия.

Для production:

- refresh token;
- HTTP-only cookies;
- ограничение времени сессии;
- logout на всех устройствах.

## Пароли

Пароли нельзя хранить в открытом виде.

Правильно:

```txt
password_hash
```

Использовать:

- bcrypt;
- argon2.

## API

Каждый API-запрос должен проверять:

- пользователь авторизован;
- пользователь имеет доступ к чату;
- пользователь имеет право писать в чат;
- запрос не превышает лимиты;
- файл имеет разрешённый тип;
- размер файла не превышает лимит.

## WebSocket

WebSocket нельзя использовать без проверки.

Нужно проверять:

- токен подключения;
- право читать чат;
- право писать в чат;
- лимит сообщений.

## Hermes-запросы

Backend должен ограничивать Hermes:

- лимит сообщений в минуту;
- лимит символов;
- таймаут;
- запрет дублирования;
- логирование ошибок;
- очистка чувствительных данных.

## Файлы

Для файлов:

- проверять MIME-тип;
- проверять расширение;
- ограничивать размер;
- хранить не под публичным путём;
- выдавать через защищённый endpoint;
- сканировать опасные типы.

## Логи

В логах нельзя хранить:

- токены;
- пароли;
- полные JWT;
- API-ключи;
- приватные сообщения без необходимости.

## HTTPS

Для production обязательно:

- HTTPS;
- secure cookies;
- HSTS;
- CSP;
- защита от XSS;
- защита от CSRF для cookie-авторизации.

## Минимальный безопасный MVP

Для GitHub Pages mock-MVP:

- нет токенов Hermes;
- нет backend-секретов;
- нет регистрации;
- нет файлов;
- данные хранятся только в localStorage.

Это безопасно как демо, но не как настоящий мессенджер.
